# 救命応援アプリ — プロジェクト基礎

「目の前で人が倒れた時、AEDが届いて、CPRがちゃんと行われ、助かる」
それを支えるアプリ。

---

## 1. プロジェクトの存在価値 (1行)

> **「救急車を呼ぶしかできなかった人」を「助けられる人」に変える。**

副題: **自宅で倒れたら助からない、を変える。AEDが届けば助かる命がある。**

---

## 2. データに基づく示唆 (Insights from Research)

### 2-1. ★ 救命の真のボトルネックは「現着時間」ではなく「AED使用率」

| 場所 (東京令和5年・心停止13,444人) | 1ヶ月生存率 | AED使用率 |
|---|---:|---:|
| 住宅 | **2.6%** | 8.3% |
| 介護施設 | 1.3% | - |
| 道路 | 10.2% | - |
| オフィス | 25.2% | - |
| 駅 | 25.6% | **44.4%** |
| スポーツ施設 | **46.4%** | - |

- **住宅 vs 公共で生存率18倍差**
- JAMA (2019, 233,511例): 公共AED使用率4.0% vs 住宅0.1% (**40倍差**)
- 場所による現着時間差はわずか **1分** (住宅9分 vs 公共8分) — 「公共で爆速・自宅で超遅延」は誤った認識

→ **アプリの本丸は「現着遅延」ではなく「AEDが住宅に届かない」を解く**

### 2-2. ★ 1分の差が救命オッズを6倍下げる

J-STAGE 2018 (全国Utstein): 覚知→接触1分延長で救命オッズ **0.15倍** (95%CI 0.042-0.533)
決定木カットオフ: **8.95分** (この閾値超で生存率が崖落ち)

→ アプリの設計目標は **「8.95分以内にAED+バイスタンダーCPRを成立させる」**

### 2-3. 都市部 (特に東京) のロングテール構造悪化中

| 階級 | 全国 | 東京 |
|---|---:|---:|
| 現着20分以上 | 4.2% | **16.0%** |
| 病院収容60分以上 | 17.3% | **46.7%** |
| 病院収容平均 | 45.6分 | **66.9分** |
| 救急搬送困難事案 (週次) | 3,120件 | **1,590件 (51%占有)** |

令和元年→令和5年で:
- 現着20分以上 1.8% → 4.2% (2.3倍)
- 収容60分以上 9.3% → 17.3% (1.9倍)
- 収容120分以上 0.3% → 1.1% (3.7倍)

→ **コロナ後も戻ってない構造的悪化**。東京は突出して悪い (全国の3.3-3.6倍)

### 2-4. ★ Coaido119 (2017-2025) の失敗要因 = 両側市場の入口狭め

サービス終了の核心:
1. **iOS限定 + 18歳以上 + 医療資格/CPR講習** ← 受信側の入口三重狭め
2. **指令センター連携なし** → 公的位置づけ得られず予算化失敗
3. **無料モデルで持続不能** (Coaido株式会社 2025-03-26 解散)
4. **全国一斉に展開で密度ゼロ** (豊島区実証 → 即全国 → スカスカ)
5. **救命実績データ非公表** で投資・自治体提携の根拠欠如

→ 本アプリは Coaido119 の **逆を行く**:
- iOS+Android (PWAから入る)
- 年齢制限緩く (段階制)
- 医療資格不要層もTier3で巻き込む
- 1区から密度ファースト
- テレメトリ最初から仕込む

### 2-5. 法律で取れる範囲が明確になった

| 論点 | MVPの方針 |
|---|---|
| 119自動発信 | 不可 (法律+OS+消防庁の四重規制)。`tel:` ユーザータップ介在のみ |
| 症状トリアージ | やらない (薬機法SaMD該当リスク)、Q助に deep link |
| 医療従事者認証 | 自己申告だけはNG (民法709条で運営責任)、免許/学生証画像→手動審査 (Antaa方式) |
| 善きサマリア法 | 日本に明文なし。民法698条 (緊急事務管理) で悪意・重過失のみ責任。利用規約三層免責+ボランティア保険で補完 |
| 応召義務 | アプリ通知に対しては無し。「向かう/無理」の二択明示OK |
| 個情法 | 位置・症状情報は要請中のみ収集、終了で破棄 |
| AEDデータ | AED N@VI (CC BY 3.0、商用OK) を採用、全国AEDマップ (財団) は商用不可 |
| ストア審査 | 「医療行為」と取られない設計を徹底、PWA配信ならストア審査回避可能 |

### 2-6. 学生プールが救助者密度の鍵

学生 (医療系) を含めると追加プール約 **30万人**:
- 医学生 約6万 / 看護学生 約20万 / 救急救命士養成課程 約4千 / 薬学部生 約8万

学生は:
- 訓練機会多 (BLS/ACLS必修化)
- 時間あり (医療従事者は仕事中)
- キャンパス周辺で密集 (=局所密度確保)
- モチベ高 (実戦経験+履歴書)
- 若くて足が速い

→ **大学医学部コミュニティ起点で1区の密度を確保** が現実的なMVP戦略 (例: 本郷/信濃町/新宿/御茶ノ水/千駄木/河田町)

### 2-7. ボランティア保険が安価で組み込み可能

- ボランティア活動保険 (東京海上日動引受、社協ルート): **年30円/人**
- 死亡1,000万、賠償5,000万
- 1万人で年30万円 → MVP予算内で組み込み可能
- ★ 救命行為が対象に含まれるかは社協確認要 (Architect段階)

## 3. コアコンセプトと位置づけ

### 3-1. ポジショニング

```
[主] 119通報   ← 発見者の義務、これは絶対。アプリは邪魔しない
  ↓
[副] 本アプリ  ← 119到着までの数分を支える
              ① AEDを届ける (最重要、住宅の弱点を解く)
              ② バイスタンダーCPRを支える (音声LLMガイド)
              ③ 医療系学生・医療者を呼ぶ (補助)
```

### 3-2. アプリは「医療行為を提供しない」

- 「人を呼ぶ + 道具を運ぶ + 行動を支える」サービス
- 「症状判定」「病名予測」「緊急度トリアージ」はやらない
- これにより薬機法SaMD非該当を維持、ストア審査も通る

### 3-3. コアターゲット

| 軸 | 設定 |
|---|---|
| **エリア** | 東京 1区から (例: 渋谷区/文京区) → 密度70%達成 → 大阪・名古屋・福岡 |
| **発生場所** | 住宅 (戸建88% + 共同住宅12%) を最優先、公共空間は副次 |
| **疾患** | OHCA (院外心停止) 心原性 = AEDが効くケース。窒息・脳卒中等は将来 |
| **発見者** | 一般人 (家族・同居人・通行人)、医療知識ゼロを前提 |
| **救助者プール** | 3層構造 (下記) |

### 3-4. 救助者の3層モデル

```
Tier 1: 医療従事者
        免許証画像→手動審査
        役割: 現場判断 + CPR交代 + 高度補助

Tier 2: 医療系学生 (医・看・救命士養成課程・薬・歯・PT/OT)
        学生証画像→手動審査
        役割: 現場到着 + CPR + AED補助
        ★ MVPの主力プール

Tier 3: AED運搬役 (講習修了者・施設管理者・近隣住民)
        修了証画像 or 身分証→手動審査 (緩く)
        役割: AEDを現場に届ける
        ★ 最重要、これが日本のOHCA救命率の鍵
```

通知は3層全員に飛ぶ。優先表示は1, 2, 3の順だが、現場到着順で受け入れ。

### 3-5. 「呼ぶ」より「変える」

```
Before: 救急車を呼ぶしかできない一般人
After:  AEDが届き、CPRが正しく行われ、助かる現場
```

## 4. 実現する機能 (MVP)

### 4-A. 発見者を「動かす」 ★ コア

| 機能 | 詳細 |
|---|---|
| SOS発火 | 巨大赤ボタン1個、3秒スワイプキャンセル猶予 |
| 症状4アイコン選択 | 呼吸なし / 意識なし / 出血 / その他 (事実選択のみ、トリアージ不可) |
| **★ 音声LLM CPRガイド** | JRC公式CPRガイドライン準拠、メトロノーム100-120BPM、視覚に依存しない (両手は患者に)、励まし音声 |
| 119コール導線 | 3秒スワイプ → `tel:` 起動 (邪魔しない、オプション扱いだが目立たせる) |
| AED マップ | AED N@VI 連動、最寄り3件表示、自分が走る選択肢も提示 |

### 4-B. AEDを「届ける」 ★ 最重要

| 機能 | 詳細 |
|---|---|
| **★ AED運搬通知** | 半径400m内の Tier3 救助者に「AEDをこの住所まで運んで」通知 |
| AED運搬役の追跡 | 発見者画面に「AEDあと2分10秒」表示、リアルタイム位置共有 |
| 運搬3択UI | 救助者通知時に「AED持って向かう / 手ぶらで向かう / 無理」 |

### 4-C. 救助者を「呼ぶ」

| 機能 | 詳細 |
|---|---|
| Tier別通知配信 | 半径400m → 応答ない場合自動拡大 |
| ロック画面即応答 | アプリ起動不要で「向かう/無理」、Web Push (PWA) / Critical Alert (将来ネイティブ) |
| 双方向位置共有 | 要請中のみ、終了で即破棄 |
| 到着推定 | Maps deep link でルート計算、ETA表示 |

### 4-D. 両者の責任を「守る」 (土台)

| 機能 | 詳細 |
|---|---|
| 行動ログ自動保存 | GPS/タイムライン/応答ログ → 善意救護の自衛資料 |
| 規約三層免責 | ストア説明文 / 起動オンボーディング / アプリ内ヘルプ |
| ボランティア保険 | 東京海上日動・社協ルート、年30円/人で団体加入 |
| 引き継ぎUI | 救助者→医療者→救急隊の段階的ハンドオフを画面で誘導 |
| キャンセル・撤退の罪悪感ゼロ設計 | 「来れない」「無理」もOK、心理的安全 |

### 4-E. 救命結果を「測る」 (テレメトリ)

| 機能 | 詳細 |
|---|---|
| 発火件数 / 通知到達率 | サーバ集計 |
| 応答率 / 現着率 | 救助者の応答ログ |
| AED運搬成功率 | 運搬完了マーク |
| ROSC寄与・神経学的予後 | 救急隊との接続後フォロー (将来) |
| 公表ダッシュボード | 月次/四半期で公開、自治体提携営業の根拠 |

→ Coaido119 が公表できなかった数字を **最初から仕込む**

## 5. 画面遷移の主シナリオ

```
[ホーム]
  画面の60%が真っ赤なSOSボタン1個
        ↓ タップ
[3秒キャンセル猶予]
  カウントダウン、画面長押しでキャンセル
        ↓ 自動進行
[症状4択]
  ┌─呼吸なし─┐ ┌─意識なし─┐
  └──────────┘ └──────────┘
  ┌──出血──┐ ┌─その他──┐
  └──────────┘ └──────────┘
  下: 「119を呼ぶ →」スワイプ
        ↓ どれか選択
[同時並行で発火]
  ├─ 119コール (ユーザースワイプで `tel:`)
  ├─ サーバ送信 (位置+症状+現場プロファイル)
  ├─ Tier3 (AED運搬) 通知配信
  └─ Tier1+2 (現場到着) 通知配信
        ↓
[救助モード画面]
  ┌────────────────────────┐
  │ 🟢 AEDあと 2分10秒      │ ← 上 (最重要モチベ)
  │ 🔵 救助者あと 3分30秒    │
  ├────────────────────────┤
  │   ♥ 押して ♥ 押して      │ ← 中央 大 (CPR音声+ビジュアル)
  │  「ターン、ターン...」     │   メトロノーム100bpm
  │   音声: 「続けて、いい感じ」│
  ├────────────────────────┤
  │ 119を再度呼ぶ            │ ← 下 (情報)
  │ 場所を確認: マンション3F │
  └────────────────────────┘
        ↓ AED到着
[AED装着ガイド]
  電源ON → パッドの貼り方 → 音声指示に従う
        ↓ 救助者到着
[引き継ぎ画面]
  「医学生さんが到着しました」
  ┌─引き継ぐ─┐ ┌─続ける─┐
        ↓ 引き継ぎ
[救助モード継続 (バトン)]
  発見者: 「あなたの行動は命を救った」+ 続けてサポート
  救助者: CPR/AEDガイド継承
        ↓ 救急隊到着
[ハンドオフ画面]
  「救急隊員にお渡しください」
  → 「引き継ぎ完了」タップ
        ↓
[終了]
  感謝表示 + ログ保存 + フィードバック
```

## 6. 情報提示の哲学 (パニック時UI、原則)

| 原則 | 具体 | 根拠 |
|---|---|---|
| 画面の60%は主要素1つ | SOS / 症状 / CPR音声 のどれか1つ | Cognitive Tunneling |
| テキスト12文字以下 | 「押して」「呼吸なし」 | パニックで文章は読めない |
| フォント24pt+、コントラスト7:1 | 黒地・白文字・赤アクセント | WCAG AAA 緊急UI |
| タップ領域88pt+ | iOSデフォの2倍 | アドレナリンで指先精度低下 |
| 1画面1〜3選択 | 多くて4 | Hick's Law |
| 音声優先、視覚副次 | CPRは声で誘導、画面は確認のみ | 両手は患者にあるから |
| キャンセルは隠す or 長押し | 緊急画面に「戻る」は無し | 誤タップ防止 |
| 進捗常時表示 | 「AEDあと2分」「救助者あと3分」 | 不確実性を消す |
| やめても罪悪感を残さない | 「お疲れ様、誰かが対応します」 | 心理的安全 |

## 7. やらないこと (スコープ外、明記)

| やらない | 理由 |
|---|---|
| 症状トリアージ・緊急度判定 | 薬機法SaMD該当リスク、Q助 (消防庁公式) に deep link |
| 119指令センター連携 | 個人開発スコープ外、年単位の協議が必要 |
| 自動119発信 | 法律+OS+消防庁の四重規制で不可 |
| 病名予測・診断 | 同上 |
| ライブ映像中継 | v1.5以降 |
| 全国一斉運用 | Coaido119の失敗。1区密度ファースト |
| 医療従事者の自己申告のみ運用 | 民法709条リスク、書類審査必須 |
| AED自前DB構築 | AED N@VI を使う、自前は鮮度維持不能 |
| 訓練・教育機能 | 別アプリ (Q助・日赤) と棲み分け、本アプリは「実戦投入装置」 |
| 課金・有料機能 | MVPはPoC、商用化は別案件 |

## 8. 技術スタックの方針 (議論中、Architect で確定)

ユーザー意思: **B (Expoネイティブ) を選択**。ただしリサーチ後に再考の余地あり:

| 案 | 強み | 弱み |
|---|---|---|
| A: PWA (Next.js等) | iOS/Android即対応、ストア審査回避、デプロイ即時 | バックグラウンド位置取得制限、ロック画面通知の即時性弱 |
| B: Expo (React Native) ★現在選択 | フル機能、ストア配信、Push確実 | EAS Build必要、ストア審査リスク |
| C: 両方並行 | 全部取れる | 工数2倍 |

★ **CPR音声LLMガイド・バックグラウンド位置共有・即時Push** はネイティブが有利
★ **デプロイ早さ・iOS/Android同時** はPWAが有利

→ **MVPはB (Expo) で固定、後日PWAも検討の余地** が現実解。ただし「寝てる間にプレビュー」要件はユーザー判断で外れた。
