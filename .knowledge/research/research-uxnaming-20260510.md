## A-2. 重要発見 (★)

### ★1. 公的指令連携 vs. 自走の差が成否を分けている
- 海外で成功している3つ (PulsePoint / GoodSAM / myResponder) は **すべて公的緊急指令 (911/999/995) と連携し、ディスパッチャーがアプリの発火源**。市民が直接発火するモデルではない。
- Coaido119 (日本・終了) は **市民発火型**。サービス終了の正確な理由は公開されていないが、消防庁・自治体との公的連携が薄かった構造的な弱さは設計上の示唆として大きい。
- 仏のStaying Alive/SAUV LifeもSAMU 15(救急医療指令)経由で発火。

→ **設計含意:** 完全な市民発火型MVPで作っても、誤報・悪用リスクと法的責任で詰む可能性が高い。最低でも 119発信を必須トリガーに同期させる (タップ→119発信＋同時にアプリSOS) アーキテクチャは設計初期で固めるべき。

### ★2. 医療従事者認証は3層あり、レベルを決める必要がある
1. **自己申告レベル** (PulsePoint): CPR受講チェックボックスのみ。手軽・大量登録◎、信頼性△
2. **書類検証レベル** (GoodSAM): 身分証アップロード+組織側で承認。中庸
3. **国家資格連携レベル** (日本ならHPKI / マイナンバー): 厚労省のHPKI(保健医療福祉分野公開鍵基盤)が医師・薬剤師を電子証明する基盤として既に存在。マイナポータル経由でも申請可能。

→ **設計含意:** MVPは2を採るのが現実的。HPKI連携はAPI公開状況の追加調査が必要 (MEDIS/日医発行・APIは現在B2B用途中心で個人アプリへの公開は不確定)。

### ★3. AEDオープンデータは整備されている
- **日本救急医療財団 全国AEDマップ** (qqzaidanmap.jp) が公的なマスタ。
- **AED N@VI** (aed-navi.jp) が **オープンデータとして再配布**、他マップとの連携を意識したフォーマット。

→ **設計含意:** AED DBを自前で作る必要なし。AED N@VI のオープンデータを定期取り込みでMVPは十分。

### ★4. 通知半径の参考値: 400m (Singapore myResponder)
研究論文ベースで400m半径が採用されている。日本の都市部でも同等が妥当。

## A-3. 学べる設計パターン (アプリ別Top3)

**PulsePoint:**
1. ディスパッチャー側のCAD (Computer Aided Dispatch) 統合APIで自動発火 (人間の判断介在不要)
2. AEDアイコンをタップすると営業時間・写真・施設名を表示 (現場で「使えるか」即判断できる)
3. 一般公開アラート (火災等) と CPR alert を分離 (CPR alertは responder のみ受信)

**GoodSAM:**
1. NHS Single Sign-Onで本人認証を国の認証基盤に委譲 (自前で資格確認しない)
2. ライブストリーミング機能 (現場映像をディスパッチャーに送信)
3. RCUK (英蘇生協議会) のオンラインCPR訓練を内蔵し、訓練修了後に登録可能化

**myResponder:**
1. 半径400mで発火 → 他地域responderに無関係通知を送らない
2. CFR (Community First Responder) を「個人」「組織 (会社単位で登録)」で分けて登録できる
3. 火災・心停止両対応 (1アプリで複数災害)

**Staying Alive / SAUV Life:**
1. 「訓練済み」「未訓練」をUI上で区別、未訓練でも電話越し指導付きで参加可能
2. AEDの「追加・修正・検証」をユーザーが行えるクラウドソーシング (40万台登録)
3. 動画チュートリアル内蔵 (心停止認知 / CPR / AED使用)

# B. パニック状態UI/UX ベストプラクティス

## B-1. 認知科学の前提

| 現象 | 内容 | 設計への影響 |
|---|---|---|
| **Cognitive Tunneling** (視野狭窄) | 注意が画面中央に強く偏る | 端のメニュー・タブは見えない |
| **微細運動能力の低下** | アドレナリンで指先精度が下がる | 標準44ptタップは誤タップ多発 |
| **Hick's Law (パニック増幅)** | 選択肢増 → 決定時間が対数増 → フリーズ | 1画面1-3選択肢が上限 |
| **読字能力低下** | 文章は頭に入らない | アイコン+短い動詞 |

## B-2. 「やってはいけない」UI 5個

1. **小さなタップターゲット (44pt以下)** — 震える指で誤タップ
2. **ネストされたメニュー階層** — 「メニュー>カテゴリ>項目」を辿れない
3. **専門用語・長文** — 「意識レベル確認」ではなく「呼びかけて反応するか?」
4. **確認ダイアログ** — 「本当に送信?」のポップアップは遮蔽物として認識され阻害
5. **キーボード入力フォーム** — 住所・氏名入力は事実上不可能。事前登録で済ませる

## B-3. 「やるべき」UI 10個

1. **ワンタップ主要アクション** — SOS・119を画面中央に巨大配置
2. **超強コントラスト 7:1以上** (WCAG AAA) — 屋外直射日光・煙下で視認
3. **アイコン+イラスト主体** — 心臓・脳・出血を絵で
4. **触覚+音声フィードバック** — 操作受付をバイブ+「119にかけています」音声
5. **Progressive Disclosure極致** — 「今この瞬間の1問」だけ表示
6. **カウントダウン自動実行** (iOS SOS流) — 3-5秒後に自動発信、その間にキャンセル可
7. **ステータス常時表示** — 「救急車到着まで6分」で安心感
8. **広域タップターゲット** — 88pt×88pt以上、視覚的なボタン枠より大きい当たり判定
9. **物理ボタン代替経路** — 音量ボタン長押し等、画面を見ずに発火
10. **音声ガイド常時** — 画面凝視できない前提で「胸の真ん中を強く速く30回」を読み上げ

## B-4. 具体数値スペック

| 項目 | 推奨値 | 根拠 |
|---|---|---|
| **最小フォントサイズ** | 20-24pt以上 | WCAG AAA + Apple HIG (本文用) |
| **コントラスト比** | 7:1 以上 | WCAG 2.1 AAA |
| **タップターゲット** | 88pt × 88pt 以上 | Appleの44pt推奨の2倍 (パニック想定) |
| **物理サイズ** | 15-20mm 以上 | 成人指腹で確実押下 |
| **1画面選択肢数** | 1-3個 | Hick's Law + Miller 7±2 のパニック減衰版 |
| **カウントダウン** | 3-5秒 | iOS Emergency SOS実装値 |
| **通知到達期待** | 60秒以内 | 心停止1分=救命率7-10%減 (Coaido記事) |

## B-5. 症状選択を「アイコン+大絵」で行う実例

- **Q助 (消防庁・大阪大学医学部附属病院)** — 大型イラスト+「はい/いいえ」2択。子供・高齢者対応。最も参考にすべき。
- **日本赤十字社「応急手当アプリ」** — AED使用を音声+大型アニメで誘導。テキスト最小化。
- **iOS Emergency SOS** — スライダー+カウントダウンで誤発火と確実発火を両立。
- **大阪市消防局「救命サポートアプリ」** — 「成人」「子供」「乳児」の3ボタンのみで動画振り分け。

→ **設計含意:** Q助のUIをそのまま参考にしつつ、トリアージ部分はQ助に飛ばすのも選択肢。

# C. 命名 (アプリ名再考)

## C-1. 前提整理

ユーザー指摘通り **GoldenHourHelper は医学的に不適切**:
- "Golden Hour" は外傷救命 (重度外傷後60分以内に決定的治療)
- 心停止のキーワードは **"Chain of Survival" / "Bystander CPR" / "First few minutes"**
- AHA/JRCのガイドラインでも cardiac arrest 文脈で Golden Hour は使わない

## C-3. ★ Researcher注意点

- 「**TASUKEL**」(Gemini初提案) は **既存商標衝突リスク高**: `taskell.jp` (介護ジョブシェアアプリ) と発音同一、`tasukel.com` (サッカー支援団体) も存在。第9類/第42類で衝突可能性、避けるべき。
- 「**Baton**」も一般語すぎて第9類で多数登録あり、避ける。
- 「**Sugu-Soba**」「**Near-Hand**」「**Inochi-Relay**」が衝突リスク・覚えやすさ両面で本命候補。
- ★ 商標調査は J-PlatPat (https://www.j-platpat.inpit.go.jp) で **第9類 (アプリ含む)・第42類 (SaaS)・第44類 (医療)** を必ず最終確認。本リサーチは「軽くチェック」レベルであり、登録前提なら弁理士相談が必要。

## C-4. 推奨 Top 3

1. **Inochi-Relay (いのちのリレー)** — 日本人の救命概念に直結、医療キーワード回避、ロゴ展開しやすい
2. **Sugu-Soba (スグソバ)** — 親しみやすさ・記憶しやすさ、商標リスク最小
3. **Near-Hand (ニアハンド)** — 国際展開も視野、商標リスク最小

# 設計含意サマリ

★最重要:
1. **市民発火型は危険**。119発信を必須トリガーに同期させるアーキを採れ。
2. **医療従事者認証はGoodSAMモデル (書類+運営承認)** をMVPで採用。HPKIは将来課題。
3. **AED DBは自前構築不要**。AED N@VI / 日本救急医療財団のオープンデータを取り込み。
4. **トリアージはQ助に丸投げ**。重複実装は誤診リスクを増やす。
5. **アプリ名は変更必須**。"Golden Hour"は心停止に医学的不適切。"Inochi-Relay"等を推奨。
6. **UI数値スペック**: フォント20pt+, コントラスト7:1, タップ88pt, 1画面1-3選択肢, カウントダウン3-5秒。
